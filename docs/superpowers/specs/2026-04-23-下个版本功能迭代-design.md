# 下个版本功能迭代设计（双主线一体化）

**日期：** 2026-04-23  
**状态：** 已定稿（待实现计划）  
**定位：** 未上线阶段的体验最优方案（C+E 双优先）

---

## 1. 设计前提（已确认）

- 目标优先级：**C（留存）与 E（传播）并列优先**。
- 项目状态：当前产品尚未正式上线，不受既有发版节奏约束。
- 决策标准：采用“**体验最优**”路线，而非最低实现成本路线。
- 实施约束：可接受较大改造（多模块重构 + 新系统整合）。
- 产品心智：采用“**双主角**”模式（个人成长与社交激励同等突出）。

---

## 2. 产品结构蓝图（双主线一体化）

### 2.1 首页信息架构

- 首页采用双主区并列：
  - `我的成长引擎`：今日目标、连续坚持、等级/心情、下一步建议。
  - `好友脉冲`：谁在坚持、谁来访、谁给我加油、可回访入口。
- 社交信息不再下沉为次级页面入口，而是与成长信息同级展示，形成“我 + 我们”的统一认知。

### 2.2 双反馈关键动作

- 用户每次完成任务时，同步触发两条反馈：
  - 个人线反馈：XP、心情、连续完成变化与语义化场景反馈。
  - 社交线反馈：可传播瞬间、好友动态更新、可被互动触点。
- 目标是将“成就感”和“被看见”绑定为一次行为的双结果，而非两个割裂流程。

### 2.3 关系域升级

- 关系页从“静态列表”升级为“今日坚持动态流”。
- 来访机制从“单次动画触发”升级为“互动场景 + 回访动作入口”。
- 增加统一收件箱入口，聚合“被加油、被来访、待回应互动”。

### 2.4 导航边界

- 核心导航保持三层：
  - `首页`：双主区与核心转化入口。
  - `关系`：关注动态、来访记录、互动入口。
  - `我的`：宠物成长、目标历史、偏好配置。
- 不引入聊天、复杂社区等重社交模块，保持轻关系产品形态。

## 3. 功能范围（Must / Should / Could）

### 3.1 Must（必须做）

1. 首页双主区重构（成长引擎 + 好友脉冲）。
2. 任务完成双反馈（成长反馈 + 社交传播触点）同步生效。
3. 关注动态流（替代仅列表展示）。
4. 收件箱中心（被加油/被来访/待回应互动聚合）。
5. 来访互动 2.0（双宠互动 + 回访入口）。
6. 分享链路强化（分享卡 -> 带参落地 -> 关注/首启 -> 关系沉淀）。
7. 成长可解释性增强（XP、streak、心情变化可理解）。
8. 云端风控闭环（奖励云发放、频控、幂等、反刷）。

### 3.2 Should（应该做）

1. “今天谁在坚持”排序策略。
2. 互动动作多样化分发策略。
3. 弱网与低性能降级策略。
4. 转化关键节点文案优化（关注、回访、首次互动）。

### 3.3 Could（可选做）

1. 周进度回顾卡（个人 + 关系双维）。
2. 连续互动轻称号。
3. 首页“下一步建议”轻策略增强。

### 3.4 明确不做（本版边界）

- 聊天/评论流。
- 双向好友确认关系。
- 公会/组队/多人实时联机。
- 全服榜单竞争系统。
- 重内容社区化模块。

## 4. 核心流程与失败兜底

### 4.1 首启流程（建立双主线认知）

1. 选宠物 -> 起名 -> 选首目标。
2. 完成首任务后立即展示双反馈（成长变化 + 可传播瞬间）。
3. 用户可选择即刻分享或稍后分享，不强制打断主流程。

### 4.2 日常打开流程（先看状态，再做动作）

1. 首页同时呈现成长状态与好友脉冲状态。
2. 用户可从任一主区进入任务执行或关系互动。
3. 保持双入口并行，不要求固定线性路径。

### 4.3 任务完成流程（版本核心）

1. 完成任务 -> 动画与数值反馈。
2. 同步生成社交资产（动态更新、分享卡可用、脉冲可见）。
3. 若用户当下不分享，保留“稍后处理”入口，避免价值损失。

### 4.4 关系回访流程（把互动转成留存）

1. 收件箱出现新事件（被加油/被来访）。
2. 一键回访触发双宠互动。
3. 回访后可继续加油或返回任务主线，形成循环。

### 4.5 异常与兜底策略

- 分享链路失败：保留应用内二次分享入口。
- 社交冷启动：好友脉冲区降级为个人建议与系统激励提示。
- 云函数失败：互动可见、奖励延迟发放并明确提示。
- 频控命中：给出可理解提示（如“今日已加油，明日可继续”）。
- 低性能设备：自动短演与低特效模式，优先主链路可用。

## 5. 技术与模块重构方案

### 5.1 页面层职责重划

- `HomePage` 仅负责布局编排与事件分发。
- 新增首页域组件：
  - `GrowthPanel`
  - `SocialPulsePanel`
- 页面层不再承担成长演算、关系聚合、分享资产生成等业务细节。

### 5.2 领域服务拆分

- `growth-service`：目标、XP、streak、心情演算与解释。
- `social-service`：关注关系、动态流、加油、来访能力。
- `moment-service`：任务完成瞬间资产生成（分享卡、摘要）。
- `inbox-service`：收件箱聚合、已读/待回应状态管理。

每个服务只处理单一领域职责，通过清晰接口通信，避免跨域耦合。

### 5.3 状态管理策略

- 首页统一状态快照：
  - `growthState`
  - `socialState`
  - `inboxState`
- 事件驱动更新关键状态：
  - `TASK_COMPLETED`
  - `CHEER_RECEIVED`
  - `VISIT_RECEIVED`
  - `FOLLOW_CHANGED`
- 目标是消除散点更新导致的状态不一致。

### 5.4 云函数能力边界

- 用户成长域：`upsertUserProfile`、`getUserSummary`
- 关系域：`followUser`、`unfollowUser`、`listFollowing`、`getFriendFeed`
- 互动域：`sendCheer`、`recordVisitInteraction`、`listVisitorsToday`
- 收件箱域：`getCheerInbox`（以及来访聚合接口）
- 分享域：`resolveSceneToken`（或等价 token 解析）

统一响应结构、错误码策略与幂等语义，降低前端分支复杂度。

### 5.5 重构落地顺序

1. 先抽服务接口，不改视觉。
2. 上线首页双主区骨架并迁移旧逻辑。
3. 接入任务完成双反馈。
4. 打通收件箱与回访链路。
5. 清理旧代码与重复状态源。

## 6. 验收指标、里程碑与风险

### 6.1 验收指标

#### C（留存）

- 有关系互动用户组 D7 高于无互动用户组。
- 被互动后 24h 回访率高于基线。
- 连续 3 天“任务完成 + 至少 1 次互动”用户占比持续提升。

#### E（传播）

- 分享回流率（打开带参页 / 发起分享 UV）提升。
- 带参落地页到关注/首启转化达标。
- 新用户首日完成“首任务 + 至少一次关系触达”的比例达标。

#### 体验质量

- 用户可清晰理解首页双主区语义。
- 用户能感知任务完成时的双反馈。
- 弱网与低性能场景下主链路不中断。

### 6.2 里程碑

- `M1`：架构就位（服务拆分 + 状态容器 + 首页骨架）。
- `M2`：闭环打通（完成双反馈 + 分享回流 + 关注沉淀）。
- `M3`：回访强化（收件箱 + 来访互动 + 回访入口）。
- `M4`：体验打磨（文案、动效、降级、风控收口）。

### 6.3 主要风险与应对

- 大改回归风险：分阶段迁移与开关控制。
- 社交冷启动不足：无好友状态降级体验与引导强化。
- 云侧复杂度上升：统一错误码、幂等、频控规则。
- 性能波动：动作与资源懒加载、分层降级。
- 范围膨胀：严格执行 Must/Should/Could 与不做清单。
