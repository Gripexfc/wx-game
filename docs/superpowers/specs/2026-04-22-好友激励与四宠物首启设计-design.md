# 好友激励与四宠物首启设计（MVP）

## 1. 背景与目标

当前产品核心循环是“个人承诺 -> 完成反馈 -> 养成成长”，缺少“人与人”关系链，导致中后期留存主要依赖单机自律。  
本设计引入低成本、可快速上线的好友激励机制（单向关注）和四宠物首启选择，目标是在个人开发可承受范围内，提升用户粘合度、次日/7日留存和长期回访。

## 2. 设计原则（腾讯产品经理视角）

- 轻社交优先：先做“看见别人 + 被别人看见 + 轻互动”，不做重社交负担。
- 低门槛闭环：分享可回流、关注可沉淀、互动可复访。
- 反作弊前置：所有成长增益必须服务端计算，不信任客户端上报数值。
- 成本可控：首期使用微信云开发，避免自建服务器与运维复杂度。
- YAGNI：MVP 不做聊天、工会、双向好友确认、全服复杂榜单。

## 3. 北极星与关键指标

### 北极星
- 人均每周“好友激励触达次数”（查看好友进度 + 加油互动）

### 关键指标（MVP）
- D1 留存提升：+3% ~ +6%
- D7 留存提升：+2% ~ +4%
- 分享回流率（打开带参页 / 发起分享 UV）：>= 12%
- 关注转化率（打开带参页后关注）：>= 25%
- 好友互动率（有关注关系用户中发生加油行为）：>= 30%
- 7 日内至少 3 次好友互动用户占比：>= 20%

## 4. MVP 范围

### 4.1 好友激励（单向关注）
- 用户可通过两类入口触达好友关系：
  - 分享卡片/二维码进入（带分享者 uid query 参数）
  - 微信好友位进入（例如会话卡片、最近使用、好友转发入口），进入后展示“好友动态页”
- 落地页提供“一键关注”能力，建立单向关系（A 关注 B，B 不需确认）。
- 关注后可查看对方公开成长摘要：
  - 今日完成数
  - 连续完成天数
  - 当前等级（或成长百分比）
- 每日轻互动“加油”：
  - 对同一好友每天最多 1 次
  - 对方获得小幅增益（如 +2 心情或 +1~2 XP，服务端统一配置）
- 好友来访互动（MVP 轻量版）：
  - 当用户从好友入口访问时，触发“双宠互动演出”（例如击掌、并肩跳、贴贴转圈）
  - 演出不改变核心战斗值，仅提供轻奖励与情绪反馈（如当天首次来访 +1 互动币）
  - 动作池扩展为三类：加油类 / PK 类 / 轻嘲讽类
    - 加油类：击掌、鼓掌、递小旗、并肩冲刺
    - PK 类：短跑比拼、跳跃对决、连招挑战
    - 轻嘲讽类：鬼脸、假装领先、转身挑衅（仅表演，不扣对方数值）
  - 不同动作对应不同反馈文案和演出镜头，保证“每次来访有新鲜感”

### 4.2 四宠物与首启流程
- 首次进入流程：选择宠物皮肤（4 选 1） -> 输入名字 -> 选择首个目标。
- 存档字段新增 `petVariantId`。
- 本期不考虑历史迁移（当前版本尚未正式上线）。

### 4.3 明确不做
- 双向好友确认
- 聊天/评论
- 全服排行榜（先做“我的关注”列表对比）
- 复杂公会/组队玩法
- 复杂多人实时联机

## 5. 闭环设计（可增长）

### 5.1 核心增长闭环
1. 用户完成任务触发“可分享时刻”  
2. 用户分享给微信好友  
3. 好友通过带参链接或好友位入口进入并关注  
4. 来访触发双宠互动演出，形成“即时情绪奖励”  
5. 关注关系沉淀后，用户每日看到好友动态并加油  
6. 双方获得“被关注/被加油/被来访”的情绪与成长反馈  
7. 触发次日回访与再次完成任务  

### 5.2 粘合度增强点
- 情绪价值：被加油反馈（文案 + 小动画 + 数值浮层）
- 关系牵引：关注页展示“谁今天在坚持”
- 轻承诺机制：可选“本周一起完成 X 次”（后续版本）
- 低负担社交：不要求双方确认，不要求即时聊天
- 来访仪式感：好友进入时触发双宠互动，强化“被看见”体验
- 连续互动钩子：连续 N 天互相加油可解锁称号/限定互动动作（后续版本）
- 周期目标钩子：每周共同目标进度条（例如“双人 20 次完成”）维持长期粘性
- 多样化刺激：PK/加油/轻嘲讽混合分发，形成轻竞技 + 轻社交的复访动机

## 6. 技术架构（微信云开发）

### 6.1 总体方案
- 客户端：微信小游戏前端
- 后端：微信云开发（云函数 + 云数据库 + 安全规则）
- 鉴权：基于小程序用户身份（openid）作为用户唯一标识

### 6.2 职责边界
- 客户端负责展示、触发动作、上报行为请求
- 云函数负责鉴权、频控、关系写入、成长增益发放
- 云数据库保存可公开摘要、关系、互动流水

### 6.3 为什么不自建服务器（MVP）
- 个人开发阶段，自建的成本在运维、安全、备案、可用性上显著更高
- 云开发足够支撑首期用户规模与玩法验证

### 7.3 cheer_logs

### 7.4 visit_logs

## 8. 安全与风控

- 所有奖励发放仅在云函数中执行；客户端禁止直接写入成长值。
- 云函数校验（**按能力区分，避免“凡互动必关注”的误读**）：
  - `sendCheer`：**必须**已存在「关注者=当前用户、被关注者=对方」；否则拒绝（与第 15.3 节一致）。  
  - `recordVisitInteraction`（双宠来访演出）：**不要求**访问前已关注，与「分享/好友位可先进页再关注」一致；频控、幂等、自访与动作分发仍全在云侧校验。  
  - 是否达到各类互动的每日次数 / 满场短演等上限。  
  - 是否重复请求（幂等）。
- 来访互动校验：
  - 来访奖励仅当天首次有效
  - 同一主客对重复触发演出不重复发奖
- 对异常行为（高频请求、批量互刷）保留审计日志字段。
- 互动体验保护：
  - 轻嘲讽支持设置中关闭（关闭后仅分发加油/PK）
  - 同一主客对中轻嘲讽每日最多触发 1 次，避免负反馈堆积
  - 未成年模式默认关闭轻嘲讽语料，仅保留中性互动动作

## 9. 关键用户路径

### 9.1 新用户首启
选皮肤 -> 起名 -> 选首目标 -> 进入首页 -> 完成首个任务 -> 可分享

### 9.2 被分享拉新
点击分享卡片 -> 带参进入 -> 展示分享者摘要 -> 一键关注 -> 开始自己的首启流程

### 9.3 日常复访
进入首页 -> 看到关注好友进度摘要 -> 给好友加油 -> 完成自己的任务 -> 再次分享/互动

### 9.4 好友位来访
从微信好友位进入 -> 展示“好友正在坚持” -> 触发双宠互动演出 -> 发起加油或关注 -> 回到个人任务页

### 9.5 双宠互动分发规则（MVP）
- 互动类型分发：加油 50%、PK 35%、轻嘲讽 15%。
- 首次来访优先加油类，保证第一感知为正向反馈。
- 同一主客对每日最多触发 3 次完整演出，第 4 次起降级为简短动作。
- 同一互动动作 24 小时内不重复触发（同一主客对），提升多样性。

## 10. 验收标准（闭环与粘合）

### 10.1 功能验收
- 首启可完成“4 选 1 皮肤 + 名字 + 首目标”
- 带参进入可识别分享者并发起关注
- 好友位进入可正确展示好友动态入口
- 关注列表可正确展示公开摘要
- 每日加油次数受限并可正确发放奖励
- 来访双宠互动可正常触发且奖励不重复发放
- PK/加油/轻嘲讽三类动作均可触发，且符合分发与频控规则

### 10.2 闭环验收
- 从“任务完成”到“分享回流”再到“关注沉淀”的链路**可线下走通或后续用运营数据/问卷抽查**（**MVP 不实现埋点 SDK**）。
- 任一中间节点失败有兜底（如关注失败重试、云函数错误提示）

### 10.3 粘合度验收
- 关注用户组的 D1/D7 显著高于未关注组（统计显著性可后续评估）
- “被加油后 24 小时回访率”高于未被加油用户
- “被来访后 24 小时回访率”高于未被来访用户
- 连续 3 天发生好友互动用户的 7 日留存高于仅单次互动用户

## 11. 评估说明（无埋点 MVP）

当前版本**不接入**统计/埋点 SDK；增长与留存评估以**产品内目标完成、云库互动流水、线下体验**为主。若后续要接数据，可再启用以下**参考事件名**（非本版交付）：

- 分享/打开/好友位、关注、加油、来访互动类型、首启选宠等。

## 12. 风险与应对

- 分享链路损耗高：优化分享素材与落地文案，弱化阻塞步骤。
- 互动动力不足：提高被加油反馈的情绪价值（非纯数值）。
- 刷量风险：加强云函数频控与幂等约束。
- 皮肤选择后悔：后续支持“低成本换装”入口（MVP 可不做）。
- 好友位入口不稳定：保留分享 query 作为主链路兜底。
- 互动演出过频疲劳：限制演出触发频率，并按关系亲密度分层解锁。

## 13. 版本规划

- V1（本次）：单向关注 + 每日加油 + 好友位来访 + 双宠互动演出 + 公开摘要 + 四宠物首启
- V1.1：关注分组、周目标同行提示、更细粒度互动反馈、连续互动称号
- V2：双向关系可选、好友小队、好友榜单玩法扩展

### 15.2 云函数清单与职责

建议 **一函数一目录**；共享逻辑可复制最小 `_lib` 或后续构建合包。

| 云函数 | 职责摘要 |
|--------|----------|
| `upsertUserProfile` | 创建/更新当前用户资料（含 `petVariantId`、昵称等） |
| `getUserSummary` | 拉取当前用户或指定公开摘要（与客户端对齐字段） |
| `followUser` / `unfollowUser` / `listFollowing` | 单向关注与列表 |
| `getFriendFeed` | 我关注的人分页 + 今日是否已加油/可选展示字段 |
| `getHostPublicPage` | 进主页前公开展示；可选合并 `resolveSceneToken` |
| `resolveSceneToken` | `token` → `hostOpenId`（若独立；亦可合并进公开展示） |
| `recordVisitInteraction` | 双宠演出：roll 动作、写 `visit_logs`、首访/满 3 场短演、防刷 |
| `sendCheer` | 每日 1 次成长向加油，写 `cheer_logs` |
| `listVisitorsToday` | 主态「今日访客」聚合 |
| `getCheerInbox` | 主态「谁给我日限加油」列表 |

**统一约定**：`wx-server-sdk` 使用 `cloud.init({ env: cloud.DYNAMIC_CURRENT_ENV })`；身份以 `cloud.getWXContext().OPENID` 为准，不信任 `event` 内「自称用户 id」。

**与演出/加油边界（已定稿）**

- `sendCheer`：成长向小 buff（心情/XP 等），**同对同自然日 1 次**，仅存 `cheer_logs`；**不**使用 `interaction_actions` 动作池。  
- `recordVisitInteraction`：双宠演出与 `visit_logs`；动作来自 **表驱动** `interaction_actions`（或云函数内等价 CONFIG）；**不与 `sendCheer` 共享「每日一次」**——今日已 cheer 仍可有 visit；**成长数值不重复从 visit 发**（首访/互动币等按本设计限额）。

### 15.3 核心产品/技术决议（已关闭）

- 后端：**微信云开发**（云函数 + 云库 + 安全规则），MVP 不自建服。  
- 关系：**单向关注**；入口 **分享 query + 好友位**（`entrySource` 区分）。  
- 未上线用户：**不做**老数据迁移。  
- `getHostPublicPage`：**允许未登录**访问公开展示（`isFollowing` 为 false）。  
- `sendCheer`：**必须已关注**（无关系则 `NOT_FOLLOWING`）。  
- 错误处理：重复业务请求优先 **`success: true` + `idempotent: true`**，避免与 `ALREADY_*` 失败态混用。  

### 15.4 错误码与统一响应

**成功**

```ts
{ success: true, data?: unknown, idempotent?: boolean }
```

**失败**

```ts
{ success: false, code: string, message?: string }
```

`code` 全大写下划线。常用枚举：

| 分类 | `code` |
|------|--------|
| 通用 | `UNAUTH`, `INVALID_PARAM`, `RATE_LIMITED`, `INTERNAL` |
| 用户/主页 | `HOST_NOT_FOUND`, `HOST_DISABLED`（可选） |
| 关系 | `ALREADY_FOLLOWING`（可幂等成功）, `NOT_FOLLOWING`, `FOLLOW_SELF` |
| 访问/分享 | `VISIT_SELF`, `INVALID_OR_EXPIRED_TOKEN` |
| 加油 | `CHEER_SELF`, `ALREADY_CHEERED`（建议同幂等成功二选一、全项目统一） |
| 可选 | `FEATURE_DISABLED` |

若 HTTP 包装：401/400/429/500 与上表可映射，以团队网关规范为准。

### 15.5 分享 `scene` / `token`

- 不在 `scene` 中长期暴露明文 `openId`。  
- **MVP 方案**：集合 `share_links`：`token`（短随机串）→ `hostOpenId`，带 `expiresAt` / `revoked`；或 **简化** 为 `users.publicLinkToken` 单用户单 token 并支持重置。  
- 云函数解析后业务侧仍带 `entrySource: share_query | friend_slot | ...`。

### 15.6 客户端 `services/cloud` 与 App 状态

**建议目录**（在现有 `js/` 下）：

- `js/services/cloud/client.js`：单次 `init`、封装 `callFunction`、统一解析成功/业务失败。  
- `js/services/cloud/user.js`、`social.js`、`visit.js`、`cheer.js`、`share.js`：按域拆分，页面禁止散落直连 `callFunction`。

**建议状态**（非埋点、仅技术路由）：`cloudReady` / `authReady|authGuest` / `onboardingPending|onboardingDone`；首启未完成不进入主玩法。

**真机与开发者工具**：联调**以真机/体验版为准**；工具内 `OPENID` 与真机非同一用户；`scene` 用「编译模式」模拟。

### 15.7 双宠资源预载与弱网

- **L1**：双方基础 idle 与皮肤；公开展示后即可预载。  
- **L2**：`recordVisitInteraction` 返回 `actionId` 后加载表驱动资源；不首包下全量动作。  
- `fullPlay === false` 时走**短演**；L2 超时可用同类别 **fallback 短动效**。

### 15.8 云库补充、索引与安全原则

**集合补充（相对第 7 节可扩展）**

- `visit_logs`：可增加 `interactionCategory`（`cheer` \| `pk` \| `tease`）、`fullPlay`、`idempotencyKey` 等，与第 7 节并存时以**实现期 PR** 为准统一字段。  
- `interaction_actions`（或云函数内 CONFIG）：`actionId`、`category`、`weight`、`tease`、`durationMs`、`assetKey`、`copyKey`、`enabled`。  

**建议索引**（控制台创建）

- `follows`：`followerOpenId` + `createdAt` 降序；`followerOpenId` + `followeeOpenId` 查重。  
- `cheer_logs`：`fromOpenId` + `toOpenId` + `dateKey`；`toOpenId` + `dateKey` + `createdAt` 降序。  
- `visit_logs`：`hostOpenId` + `dateKey` + `createdAt`；`visitorOpenId` + `hostOpenId` + `dateKey`。

**安全原则**：`follows` / `cheer_logs` / `visit_logs` **默认仅云函数写**；`users` 敏感字段只云改；查询防越权（访客列表/收件箱仅本人）。
