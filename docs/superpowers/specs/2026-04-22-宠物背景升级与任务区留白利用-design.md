# 宠物背景语义化场景设计（任务驱动）

## 1. 目标与范围

当前宠物背景的问题不是“动不动”，而是“没有语义场景”，导致观感单调、记忆点弱。  
本设计将背景升级为“任务语义驱动 + 通用场景池兜底”。

目标：

1. 任务完成后，背景与宠物姿态明确表达任务语义。
2. 非语义任务或无任务状态下，背景进入高质量通用场景（而非纯色/纯渐变）。
3. 保持代码可维护，严格模块拆分，不再将背景逻辑堆在 `HomePage`。

确认范围：

- 先做 3 个专属主题：`健身 / 阅读 / 饮食`
- 再做 1 套通用兜底：`场景池（房间 / 公园 / 工作角）`
- 不符合三类任务的一律走通用兜底

## 2. 设计原则

1. **语义优先**：用户一眼能看出“刚完成了什么类型任务”。
2. **主角优先**：场景服务宠物，不压主体。
3. **即时反馈 + 稳定停留**：完成任务立刻切主题，但有停留窗口，避免频闪。
4. **纯 Canvas 实现**：阶段 1 不引入图片资源，先保证可调与轻量。
5. **模块边界清晰**：渲染、主题决策、姿态决策分层。

## 3. 主题体系

## 3.1 专属任务主题（3个）

1. `fitness_power`（健身）
   - 视觉：斜向力量光束、地台冲击纹、偏橙红高对比
   - 宠物姿态：挺胸握拳/展示肌肉
2. `read_focus`（阅读）
   - 视觉：书桌台灯、页片漂浮、偏暖白与木色
   - 宠物姿态：捧书低头、专注微动
3. `eat_vitality`（饮食）
   - 视觉：餐桌/厨房角、果蔬色块、柔和圆润高光
   - 宠物姿态：满足拍拍肚子、元气比赞

## 3.2 通用场景池（兜底）

1. `home_cozy`（房间）
2. `park_breeze`（公园）
3. `work_focus`（工作角）

规则：

- 当任务类型不在三类内，或当前无任务语义激活时，使用通用场景池。
- 通用场景采用“加权选择”，由心情档位参与权重：
  - 开心：公园权重高
  - 平稳：房间权重高
  - 专注/高完成度：工作角权重高

## 4. 主题切换策略（时序）

状态模型：

- `activeTheme`：当前主题
- `source`：`task` 或 `ambient`
- `lastTaskTheme`：最近语义任务主题
- `taskThemeExpiresAt`：语义主题过期时间
- `transitionProgress`：过渡进度 0~1

切换规则：

1. 完成三类任务之一：立即切到对应专属主题，`source=task`。
2. 专属主题保持 `45 分钟`（可配置），期间再次完成任务则刷新主题和过期时间。
3. 过期后自动回到 `ambient`，由通用场景池决策。
4. 切换使用 300~500ms 交叉淡入，禁止硬切。

## 5. 交互反馈

1. 点击宠物：轻涟漪（统一）
2. 完成任务：
   - 专属主题强化特效（例如健身震荡纹）
   - 宠物进入对应姿态动画
3. 升级：
   - 保留现有升级环
   - 不强制切换任务主题，避免语义被打断

## 6. 代码拆分设计（必须执行）

## 6.1 新增模块

1. `js/ui/pet-background/ThemeEngine.js`
   - 输入：任务完成事件、心情、时间
   - 输出：当前主题与过渡状态

2. `js/ui/pet-background/SceneRenderer.js`
   - 输入：主题、过渡进度、画布区域
   - 输出：场景绘制（房间/公园/工作角 + 三专属）

3. `js/ui/pet-background/ParticleSystem.js`
   - 输入：主题参数、事件触发
   - 输出：粒子与短时特效

4. `js/ui/pet-background/PetPoseController.js`
   - 输入：主题、事件
   - 输出：宠物姿态标识（给 `Lulu` 使用）

5. `js/ui/pet-background/index.js`
   - 对外统一导出背景系统 API，供 `HomePage` 使用

## 6.2 现有模块改动

1. `js/ui/HomePage.js`
   - 仅保留调用入口，不承载复杂背景细节
2. `js/Lulu.js`
   - 增加可选姿态接口：`setPose(poseId, duration)`
3. `js/main.js`
   - 在任务完成回调处透传“任务类型事件”给背景系统

## 7. 数据与事件

新增运行时事件（内存态，不落盘）：

- `onGoalCompleted({ goalId, tag, type })`
- `onMoodChanged({ moodValue, moodLabel })`

可选持久化（阶段 2 再做）：

- `lastTaskTheme`
- `taskThemeExpiresAt`

阶段 1 默认不持久化，重启后回到通用场景池。

## 8. 异常与边界

1. 未识别任务标签：直接回退 `ambient` 场景池
2. 高频完成任务：主题切换做最短间隔（如 600ms）避免抖动
3. 低性能设备：
   - 降粒子数量
   - 降云层数量
   - 关闭部分高阶特效

## 9. 验收标准

功能：

1. 完成健身/阅读/饮食任务后，背景与姿态均有明确区别。
2. 完成非三类任务时，进入通用场景池，不出现空白背景。
3. 主题切换有过渡，无硬切闪烁。

质量：

1. 背景复杂逻辑不再堆在 `HomePage`（代码拆分完成）。
2. 任务完成、撤销、升级主流程无回归。
3. lint/test 基线通过。

## 10. YAGNI 边界

本轮不做：

1. 主题编辑器或用户自定义场景
2. 多资源图片化大场景
3. 全任务类型的专属主题（仅三类 + 通用兜底）

## 11. 交付里程碑

1. 模块拆分骨架（ThemeEngine / SceneRenderer / ParticleSystem / PoseController）
2. 通用场景池实现（房间/公园/工作角）
3. 三专属主题与姿态实现（健身/阅读/饮食）
4. 主题切换时序与回归验证
